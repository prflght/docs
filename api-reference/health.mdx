`GET /health` returns live health scores and TVL data for every DeFi protocol Prflght monitors. Prflght pulls TVL figures from DeFiLlama and derives a `healthScore` between 0 and 100 for each venue. This data feeds directly into policy evaluation — transactions touching venues below your configured health threshold are automatically blocked.

## Endpoint

```
GET https://api.prflght.xyz/health
```

No request parameters or authentication are required.

## Response fields

<ResponseField name="venues" type="array" required>
  List of monitored DeFi protocol objects. Each entry represents one venue Prflght tracks.

  <Expandable title="venues properties">
    <ResponseField name="id" type="string" required>
      Protocol identifier string. Current values include `"jupiter"`, `"drift"`, `"orca"`, `"kamino"`, and `"raydium"`, among others.
    </ResponseField>
    <ResponseField name="healthScore" type="number" required>
      Integer from 0 to 100 representing the current health of the protocol. A score of 100 means the protocol is fully healthy. Scores below your policy's `deny_if_protocol_health_below` threshold will cause Prflght to block transactions that route through that venue.
    </ResponseField>
    <ResponseField name="tvlUsd" type="number" required>
      Total value locked in the protocol, denominated in USD. Sourced from DeFiLlama. Used to evaluate your policy's `pause_on_protocol_tvl_drop` rule.
    </ResponseField>
    <ResponseField name="status" type="string" required>
      Summary status string: `"healthy"` or `"warning"`. A `"warning"` status does not necessarily mean transactions are blocked — the exact behavior depends on your policy configuration.
    </ResponseField>
  </Expandable>
</ResponseField>

### Response example

```json
{
  "venues": [
    {
      "id": "jupiter",
      "healthScore": 95,
      "tvlUsd": 2100000000,
      "status": "healthy"
    },
    {
      "id": "drift",
      "healthScore": 42,
      "tvlUsd": 180000000,
      "status": "warning"
    }
  ]
}
```

## How this data feeds into policy

Prflght evaluates two protocol-health policy rules on every transaction:

- **`deny_if_protocol_health_below`** — if a venue involved in the transaction has a `healthScore` below this threshold, Prflght returns a `deny` decision. Set this in your agent's policy to block transactions during periods of degraded protocol health.
- **`pause_on_protocol_tvl_drop`** — if a venue's TVL drops by more than the configured fraction since your agent last ran, Prflght pauses the agent entirely until you explicitly resume it. This guards against sudden liquidity crises.

See the [Protocol Health concepts page](/concepts/protocol-health) for a full explanation of how health scores are calculated and how to configure these thresholds.

## Monitored protocols

Prflght currently tracks the following venues:

- Jupiter
- Drift
- Orca
- Kamino
- Raydium

Plus 7 additional venues. The full list is reflected in the `venues` array returned by this endpoint.

<Tip>
  Call `GET /health` before constructing your transaction to proactively check whether the venues your agent plans to interact with are healthy. This avoids a round-trip to `POST /check` for a transaction that would be denied anyway due to low protocol health.
</Tip>
